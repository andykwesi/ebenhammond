# Credit Card Fraud Detection System Using Machine Learning

## 1. Introduction

Credit card fraud is one of the major challenges facing modern electronic payment systems. As banking, e-commerce, mobile money, and digital payment platforms continue to grow, large volumes of card transactions are processed every day. Most of these transactions are legitimate, but a small number are fraudulent and may lead to financial losses, customer dissatisfaction, reputational damage, and additional investigation costs for financial institutions. Fraud detection is therefore an important area in Information and Communication Technology (ICT), because it combines data management, software engineering, machine learning, web application development, and decision support.

The purpose of this final year project is to design and implement a machine learning-based credit card fraud detection system. The system trains multiple classification models, compares their performance using fraud-sensitive evaluation metrics, selects the best-performing model, and provides a Flask web interface for single and batch transaction prediction. The project is designed as a practical ICT solution rather than only a theoretical machine learning experiment. It includes data validation, model training, artifact storage, a dashboard, prediction pages, batch CSV upload, metrics display, and prediction logging.

Credit card fraud detection is a suitable ICT final year project because it addresses a real-world problem and demonstrates the practical application of data science in financial technology. It also requires the integration of several technical areas: data preprocessing, machine learning model development, backend web programming, user interface design, testing, and system evaluation. The implemented system uses Python, Flask, pandas, scikit-learn, imbalanced-learn, HTML templates, and local file-based storage for trained models, metrics, and prediction logs.

The problem addressed by this project is the difficulty of detecting fraudulent credit card transactions in highly imbalanced datasets. In most real transaction datasets, fraudulent transactions form only a very small percentage of all transactions. For example, the public European credit card fraud dataset contains 284,807 transactions, but only 492 are fraudulent, representing about 0.172% of the dataset (Machine Learning Group - ULB, n.d.; OpenML, n.d.). This means that a simple model can appear highly accurate by predicting every transaction as legitimate, while completely failing to detect fraud. Therefore, ordinary accuracy is not enough for evaluating fraud detection systems. The system developed in this project uses metrics such as recall, precision, F1-score, ROC-AUC, PR-AUC, and confusion matrix values to provide a more meaningful evaluation.

The main aim of the project is to develop a web-based credit card fraud detection system using supervised machine learning techniques. The specific objectives are to collect or load credit card transaction data using the expected dataset schema, validate transaction records before training or prediction, train and compare several machine learning models, handle class imbalance using SMOTE where appropriate, select the best model using fraud-focused metrics, provide a dashboard for viewing system status and model performance, allow single transaction prediction through a form, allow batch prediction through CSV upload, and test the system to confirm that core workflows operate correctly.

The scope of the project is limited to a prototype fraud detection system using classical supervised machine learning methods. The implemented models are Logistic Regression, Random Forest, Support Vector Machine, and Gradient Boosting. The system expects transaction features in the format used by the public European credit card fraud dataset: `Time`, `V1` to `V28`, `Amount`, and `Class` for training data. For prediction data, the target column is not required. The system stores the best trained model in a local artifact file and stores model metrics in JSON and CSV formats. It also logs prediction activity locally. The project does not connect to a live banking system, payment gateway, or external database. It is intended for academic demonstration and prototype evaluation.

The significance of this project lies in its practical demonstration of how machine learning can support financial fraud detection. The system shows that ICT solutions can be built by combining data-driven models with usable software interfaces. It also demonstrates the importance of choosing suitable evaluation metrics for imbalanced classification problems. For students and researchers, the project provides a clear example of how machine learning concepts can be implemented in a working web application. For financial institutions, it illustrates the type of decision support system that can assist fraud analysts, although further work would be needed before real deployment.

The rest of this report is organized into five sections. Section 1 introduces the background, problem, objectives, scope, and significance of the project. Section 2 reviews literature on credit card fraud detection, machine learning methods, class imbalance, evaluation metrics, and related systems. Section 3 presents the system analysis and design, including requirements, architecture, data flow, and system modules. Section 4 explains the implementation and testing of the system. Section 5 presents the summary, conclusion, and recommendations for future work.

## 2. Literature Review

Credit card fraud detection is usually treated as a binary classification problem in which each transaction is classified as legitimate or fraudulent. The classification task appears simple, but the real-world problem is complex because fraud is rare, fraud patterns change, labels may be delayed, and the cost of different types of errors is not equal. A false negative allows a fraudulent transaction to pass as legitimate, while a false positive may inconvenience a genuine customer and increase manual review workload. Therefore, an effective fraud detection system must balance fraud detection performance with operational usability (Dal Pozzolo et al., 2018).

Machine learning has become an important approach for fraud detection because it can learn patterns from historical transaction data. Traditional rule-based systems depend on fixed rules created by experts, such as flagging unusually high transaction amounts or transactions from suspicious locations. Although rules can identify known fraud patterns, they may become less effective when fraudsters change their behavior. Machine learning models can identify relationships among features that may not be obvious to human analysts and can be retrained when new data becomes available. Lucas and Jurgovsky (2020) explain that data-driven credit card fraud detection involves not only model selection but also dataset characteristics, metric choice, class imbalance, concept drift, and sequential transaction behavior.

The European credit card fraud dataset is one of the most widely used datasets for academic fraud detection projects. It contains transactions made by European cardholders over two days in September 2013. The dataset includes 284,807 transactions, of which 492 are fraudulent (Machine Learning Group - ULB, n.d.; OpenML, n.d.). It contains `Time`, `Amount`, and 28 anonymized principal component features labeled `V1` to `V28`. The target column, `Class`, uses `1` for fraudulent transactions and `0` for legitimate transactions. The dataset is useful because it is public, real-world, and strongly imbalanced. However, it also has limitations. Since most features are anonymized, it is difficult to interpret why a particular transaction is suspicious. Also, the dataset covers only a short period, so it does not fully represent long-term fraud changes.

Class imbalance is one of the most important challenges in fraud detection. He and Garcia (2009) describe imbalanced learning as a situation where one class has far fewer examples than another. In credit card fraud detection, legitimate transactions are the majority class, while fraudulent transactions are the minority class. Standard classification algorithms may become biased toward the majority class because they can reduce overall error by predicting most transactions as legitimate. This is dangerous in fraud detection because the minority class is the class of greatest interest.

Several techniques have been proposed to address class imbalance. One approach is undersampling, where some majority-class records are removed to reduce imbalance. Another is oversampling, where minority-class records are increased. Chawla et al. (2002) introduced SMOTE, the Synthetic Minority Over-sampling Technique, which creates synthetic minority examples by interpolating between existing minority-class records. SMOTE is better than simple duplication because it adds new synthetic samples rather than copying the same minority examples. Ileberi et al. (2021) applied SMOTE in a credit card fraud detection framework and evaluated several machine learning models, showing that resampling is commonly used in this domain.

Class weighting is another imbalance-handling method. Instead of changing the dataset, it changes the learning objective by giving more importance to minority-class errors. Logistic Regression, Random Forest, and Support Vector Machine classifiers commonly support balanced class weights. In this project, class weighting is used together with SMOTE for selected models. This is appropriate because SMOTE increases the representation of fraud cases, while class weighting penalizes mistakes on the fraud class more strongly.

Logistic Regression is a common baseline model for binary classification. It estimates the probability of class membership using a linear combination of input features. It is fast, easy to train, and relatively interpretable. In fraud detection, Logistic Regression can be useful when the relationship between features and fraud is approximately linear or when transformed features already capture useful patterns. However, it may not perform well when fraud patterns depend on complex nonlinear interactions among variables.

Random Forest is an ensemble learning method introduced by Breiman (2001). It builds many decision trees and combines their predictions. Random Forest can capture nonlinear relationships and interactions between features. It is also less sensitive to feature scaling than linear or distance-based models. Bhattacharyya et al. (2011) included Random Forest in a comparative study of credit card fraud detection methods and showed that it is a relevant technique for financial fraud classification. In this project, Random Forest is used because it is strong for tabular data and can support class weighting.

Support Vector Machine is a margin-based classification algorithm that attempts to find a boundary separating classes with the largest possible margin (Cortes & Vapnik, 1995). With kernel functions such as the radial basis function kernel, SVM can model nonlinear decision boundaries. SVM can be effective for fraud detection when fraudulent and legitimate transactions are separable in transformed feature space. However, SVM can be computationally expensive on large datasets and is sensitive to feature scaling and hyperparameter selection. The system therefore applies standard scaling before training the SVM model.

Gradient Boosting is another ensemble method. It builds models sequentially, where each new learner attempts to correct errors made by previous learners. Friedman (2001) describes gradient boosting as a method for building additive predictive models. In fraud detection, boosting can be useful because it focuses on difficult cases and can model complex relationships. This project includes Gradient Boosting as a comparative model so that the system can evaluate a boosting approach alongside linear, bagging, and margin-based methods.

Evaluation metrics are very important in fraud detection. Accuracy alone is not suitable for highly imbalanced datasets because a model can achieve high accuracy while missing almost all fraud cases. The confusion matrix provides a better foundation by showing true positives, false positives, true negatives, and false negatives. Recall measures the proportion of actual fraud cases detected by the model. Precision measures the proportion of predicted fraud cases that are actually fraud. F1-score combines precision and recall. ROC-AUC measures ranking performance across thresholds, while PR-AUC focuses more directly on the minority positive class.

Davis and Goadrich (2006) explain the relationship between precision-recall and ROC curves. Saito and Rehmsmeier (2015) argue that precision-recall plots are more informative than ROC plots when evaluating classifiers on imbalanced datasets. This is highly relevant to credit card fraud detection because the positive class is rare. A model may have good ROC-AUC but poor precision when the number of false positives is large. For this reason, the current system reports PR-AUC in addition to ROC-AUC and other metrics.

Concept drift is another important issue in fraud detection. Gama et al. (2014) define concept drift as a change in the relationship between input data and target labels over time. In credit card fraud, drift may occur because customer behavior changes, fraudsters develop new techniques, and security systems influence attacker behavior. Dal Pozzolo et al. (2018) identify class imbalance, concept drift, and verification latency as major challenges in realistic credit card fraud detection. Verification latency occurs when labels are not immediately available because fraudulent transactions may only be confirmed after investigation or customer complaint.

Feature engineering is also important. Whitrow et al. (2009) studied transaction aggregation as a strategy for credit card fraud detection and found that aggregated transaction behavior can improve detection in some cases. Aggregated features may include transaction frequency, total amount over a period, average spending, or changes in customer behavior. The dataset used in this project is anonymized and already transformed using principal component analysis, so the project focuses more on model comparison and system implementation than advanced feature engineering.

From the literature, it is clear that an effective fraud detection prototype should use suitable models, address class imbalance, evaluate performance with fraud-sensitive metrics, and provide a practical interface for prediction. The system developed in this project follows these principles. It compares Logistic Regression, Random Forest, Support Vector Machine, and Gradient Boosting; applies SMOTE where appropriate; uses class weights for selected models; evaluates models using recall, precision, F1-score, ROC-AUC, PR-AUC, and confusion matrix values; and provides web pages for training, metrics, single prediction, and batch prediction. This makes the project a suitable ICT final year system because it combines research-based machine learning decisions with practical software implementation.

## 3. System Analysis and Design

System analysis involves understanding the problem, identifying requirements, and defining how the proposed system should operate. The proposed system is a credit card fraud detection web application that allows users to train machine learning models and use the selected model to classify transactions as legitimate or fraudulent. The system is designed for academic demonstration and prototype use. It does not connect to a live payment system, but it provides the main components required for a basic fraud detection workflow.

The existing problem is that manual fraud detection is slow and rule-based methods may not detect new fraud patterns. Financial transaction datasets are also highly imbalanced, making it difficult to identify fraudulent transactions using ordinary classification approaches. The proposed system addresses this problem by using supervised machine learning models trained on labeled transaction data. It also provides a user interface so that users can interact with the model without writing code.

The functional requirements of the system are as follows. The system should load a credit card transaction dataset from a CSV file. It should validate that the required columns are present and numeric. It should train multiple machine learning models. It should handle class imbalance using SMOTE where the dataset contains enough fraud examples. It should evaluate all trained models using appropriate metrics. It should select and save the best model. It should display the best model and performance metrics. It should allow users to enter a single transaction and receive a fraud prediction. It should allow users to upload a CSV file for batch prediction. It should allow users to download batch prediction results. It should record prediction activity in a local log file.

The non-functional requirements include usability, reliability, maintainability, and portability. The system should be easy to run locally by a student or examiner. The web interface should be simple and understandable. The code should be organized into modules so that data handling, modeling, prediction, and web routing are separated. The system should validate user inputs to prevent invalid data from being passed into the model. It should also store trained artifacts and metrics so that the model does not need to be retrained every time the application starts.

The system users are mainly students, lecturers, project supervisors, and demonstration users. A student can use the system to train models, test predictions, and explain how machine learning supports fraud detection. A lecturer or supervisor can use the dashboard and metrics page to assess whether the system works as intended. A demonstration user can enter sample transaction values or upload a CSV file to see the prediction workflow.

The system architecture follows a simple layered structure. The presentation layer consists of Flask templates for the dashboard, metrics page, single prediction page, batch prediction page, and base layout. The application layer consists of Flask routes that receive requests, call model or prediction functions, and return rendered pages. The machine learning layer contains functions for training, evaluation, model loading, and prediction. The data layer contains CSV datasets, saved model artifacts, metrics files, and prediction logs.

The main modules of the system are the data module, modeling module, prediction module, configuration module, and web application module. The data module validates transaction datasets and creates a synthetic demo dataset if the public dataset is not available. The modeling module builds the machine learning pipelines, trains models, evaluates performance, selects the best model, and saves metrics. The prediction module loads the trained model, validates prediction input, generates fraud probabilities, assigns labels, and logs predictions. The configuration module defines file paths and feature column names. The web application module defines Flask routes and connects the user interface to the backend logic.

The dataset design is based on the European credit card fraud dataset schema. For training, the required columns are `Time`, `V1` to `V28`, `Amount`, and `Class`. The `Class` column is the target variable, where `0` means legitimate and `1` means fraud. For prediction, the input requires `Time`, `V1` to `V28`, and `Amount`, but not `Class`. The system validates that all required columns exist and that values are numeric. If missing or non-numeric values are found, the system raises a validation error.

The model training design begins by loading and validating the dataset. The data is separated into features and target labels. A stratified train-test split is used so that both training and test sets preserve the class distribution. The system then builds four model pipelines: Logistic Regression, Random Forest, Support Vector Machine, and Gradient Boosting. SMOTE is added to selected pipelines when there are enough fraud examples. Standard scaling is applied to Logistic Regression and SVM because these models are sensitive to feature magnitude. After training, each model is evaluated using accuracy, precision, recall, F1-score, ROC-AUC, PR-AUC, and confusion matrix values.

The model selection design prioritizes fraud detection performance. The system ranks models by recall first, then F1-score, then PR-AUC, and finally fewer false positives. This ranking method reflects the fraud detection context, where identifying fraudulent transactions is more important than maximizing general accuracy. The best model is saved as a model bundle containing the model name, trained model object, feature columns, and ranking rule. Metrics are saved in JSON and CSV formats for display and review.

The prediction design uses the saved best model. For single prediction, the user enters values for all feature columns in a form. The system validates and converts the values to numeric format, sends them to the model, obtains the fraud probability, and assigns a label using a threshold of 0.5. For batch prediction, the user uploads a CSV file. The system validates the file, predicts each transaction, adds `fraud_probability`, `prediction`, and `prediction_label` columns, displays a preview, logs the predictions, and stores the result for download.

The dashboard design provides a summary of system activity. It displays the best model, important metrics, and prediction summary values such as total predictions, fraud predictions, and legitimate predictions. The metrics page displays the performance of all trained models. The batch page allows CSV upload and result download. The single prediction page allows manual transaction scoring.

Security and privacy are important considerations in fraud detection systems. The project uses anonymized public data and local files, so it does not process real customer-identifiable information. However, if the system were extended for real-world use, stronger security would be necessary. This would include authentication, authorization, encrypted storage, secure logging, audit trails, access control, and compliance with data protection regulations.

The system design is suitable for a final year ICT project because it is clear, modular, and demonstrable. It uses accepted software development principles while also applying machine learning to a real-world problem. The separation of concerns makes the system easier to maintain and explain. The design also allows future improvements, such as adding a database, using more advanced models, improving threshold selection, or deploying the application online.

## 4. System Implementation and Testing

The system was implemented using Python and Flask. Python was selected because it has strong support for data analysis and machine learning through libraries such as pandas, scikit-learn, imbalanced-learn, and joblib. Flask was selected because it is lightweight and suitable for building small to medium web applications. The implementation is organized into separate folders for the Flask application, machine learning logic, tests, data, artifacts, and logs.

The project structure includes an `app` folder for the web application, a `fraud_detection` package for machine learning and data processing, a `tests` folder for automated tests, a `data` folder for datasets, an `artifacts` folder for trained model files and metrics, and a `logs` folder for prediction logs. This organization supports maintainability because each part of the system has a clear responsibility.

The configuration file defines the base directory, data directory, artifact directory, log directory, default dataset path, model path, metrics paths, prediction log path, feature columns, and target column. The feature columns are defined as `Time`, `V1` to `V28`, and `Amount`. The target column is `Class`. This central configuration ensures that all modules use the same dataset schema and file locations.

The data module implements dataset loading and validation. If the public dataset is not available, the system can create a small synthetic demo dataset that follows the same schema. This allows the project to run during demonstrations even when the real dataset file is missing. However, the report recognizes that the synthetic dataset is only for demonstration and should not be used for final academic performance claims. The validation function checks that required columns exist, converts values to numeric format, rejects missing or non-numeric values, and confirms that the target column contains only `0` and `1` during training.

The modeling module implements model training and evaluation. It defines a function for building model pipelines and another function for training and evaluating them. Logistic Regression and SVM use `StandardScaler` because they are affected by feature scale. Logistic Regression, Random Forest, and SVM use balanced class weights. SMOTE is included when enough fraud examples are available. The models are trained on a stratified training split and evaluated on a test split. The evaluation function calculates accuracy, precision, recall, F1-score, ROC-AUC, PR-AUC, true negatives, false positives, false negatives, and true positives.

The system selects the best model using a ranking rule that prioritizes recall, then F1-score, then PR-AUC, then fewer false positives. This implementation decision supports the project objective because fraud detection should prioritize detecting fraud rather than only achieving high general accuracy. After selecting the best model, the system saves it using joblib. It also writes model metrics to JSON and CSV files so that the web application can display them.

The prediction module implements single and batch prediction. It loads the saved model bundle, validates the transaction input, extracts the correct feature columns, obtains predicted fraud probabilities, applies a threshold of 0.5, and returns prediction labels. For a single transaction, the result includes the predicted class, prediction label, and fraud probability. For batch prediction, the system returns the uploaded data with additional prediction columns. The module also logs prediction events with timestamp, source, predicted class, label, and probability.

The Flask application connects the backend functions to web pages. The dashboard route loads metrics and displays the best model and prediction summary. The training route retrains models and redirects the user to the metrics page. The prediction route supports both GET and POST requests for single transaction scoring. The batch route supports CSV upload, prediction, preview display, logging, and storing downloadable results. The download route returns the batch predictions as a CSV file. The metrics route displays all model performance results.

The user interface was implemented with HTML templates and CSS. The base template provides the common layout, while separate templates handle dashboard, metrics, batch prediction, and single prediction pages. The interface is designed to be simple and suitable for an academic project demonstration. It allows a user to access the major functions without interacting directly with the command line.

Testing was carried out using automated tests with pytest. The first test confirms that the demo dataset matches the expected schema and includes the required feature columns and target column. The second test checks that validation rejects a dataframe with missing required columns. The third test performs a training and prediction smoke test by creating a temporary demo dataset, training models, confirming that model and metrics artifacts are saved, and checking that predictions include `fraud_probability` and `prediction_label` columns.

These tests are important because they verify the core workflows of the system. Dataset schema validation ensures that invalid input is caught before training or prediction. The training smoke test confirms that the machine learning pipeline can train successfully and save artifacts. The prediction smoke test confirms that the trained model can be used to score new transactions. Although these tests do not cover every possible user interaction, they provide confidence that the main system functions operate correctly.

Manual testing can also be performed through the web interface. The user can run the training script, start the Flask application, open the dashboard, retrain models through the application, inspect the metrics page, enter a single transaction, upload a batch CSV file, preview predictions, and download prediction results. The system should show clear success or error messages depending on the action performed. Invalid inputs such as missing form values, non-numeric values, empty uploaded files, or CSV files with missing columns should produce validation errors.

The implementation meets the project objectives. It loads and validates data, trains several models, handles imbalance, evaluates models using suitable metrics, selects and saves the best model, provides a web interface, supports single and batch prediction, logs prediction activity, and includes automated tests. The implementation is also extensible because new models, metrics, data storage methods, or authentication features can be added later.

Some implementation limitations remain. The system uses local file storage rather than a database. It applies a fixed 0.5 threshold for prediction labels. It does not include user authentication. It does not perform real-time streaming prediction. It does not include probability calibration or cost-sensitive threshold optimization. It does not connect to real banking infrastructure. These limitations are acceptable for a final year prototype but should be addressed before any real-world deployment.

## 5. Summary, Conclusion and Recommendation

This project developed a credit card fraud detection system using machine learning. The system was designed as a final year ICT project that combines data preprocessing, supervised learning, imbalanced classification, web application development, and system testing. The project addressed the problem of detecting rare fraudulent transactions in a highly imbalanced dataset and demonstrated how machine learning can support fraud detection decision-making.

The system implemented four machine learning models: Logistic Regression, Random Forest, Support Vector Machine, and Gradient Boosting. It handled class imbalance using SMOTE where appropriate and used class weighting for selected models. It evaluated models using metrics that are suitable for fraud detection, including precision, recall, F1-score, ROC-AUC, PR-AUC, and confusion matrix values. The best model was selected using a ranking strategy that prioritizes recall, followed by F1-score, PR-AUC, and fewer false positives.

The system also provided a Flask web interface. The dashboard displays the best model and prediction summary. The metrics page displays model performance. The single prediction page allows manual scoring of one transaction. The batch prediction page allows CSV upload, preview of prediction results, logging, and download of predicted results. This makes the project more practical than a notebook-only experiment because users can interact with the trained model through a web application.

The literature review showed that credit card fraud detection is a difficult problem because fraud is rare, behavior changes over time, and false positives and false negatives have different costs. The literature also showed that accuracy is not enough for evaluating imbalanced fraud detection systems. Precision, recall, F1-score, ROC-AUC, PR-AUC, and confusion matrix values are more appropriate. The implemented system follows these findings and applies them in a working prototype.

In conclusion, the project successfully demonstrates the development of a machine learning-based credit card fraud detection system for an ICT final year project. The system meets its main objectives by training and comparing models, handling imbalanced data, selecting a best model, supporting prediction, displaying metrics, and testing important workflows. It provides a strong foundation for further academic work and practical improvement.

Several recommendations are proposed for future work. First, the system should be extended to use a database such as PostgreSQL or MySQL instead of local file storage. This would improve data management, auditability, and scalability. Second, user authentication and role-based access control should be added so that only authorized users can train models, upload files, or view prediction results. Third, the system should support threshold tuning so that administrators can choose a decision threshold based on business needs, such as maximizing recall or limiting false positives.

Fourth, future work should include probability calibration and cost-sensitive evaluation. In real fraud detection, the cost of missing fraud may be different from the cost of investigating a legitimate transaction. A cost-sensitive approach would make the system more realistic. Fifth, the system should be tested using chronological validation to better simulate real-world deployment, where models are trained on past transactions and tested on future transactions. Sixth, more advanced models such as XGBoost, LightGBM, CatBoost, neural networks, or anomaly detection methods could be compared with the current classical models.

Finally, future versions of the system could include better reporting and visualization. Charts showing model comparison, confusion matrices, precision-recall curves, and fraud probability distributions would make the dashboard more informative. The project could also be deployed online using a cloud platform so that supervisors and users can access it more easily. With these improvements, the prototype could evolve into a more complete fraud detection decision support system.
